Self-custodial

The wallet key is derived from the passkey and wiped after each use. Never stored, never on a server.

No backend

A client-side SDK. Bring your own RPC, bundler, paymaster, and Kora. Avok runs nothing.

Passkey-native

Face ID, Touch ID, or a fingerprint is the wallet. No seed phrase and no extension.

EVM and Solana

Smart EOAs through EIP-7702, plus a first-class Solana rail, from one API.

Two signing paths

The passkey's rpId decides where signing happens.

You select the path through configuration. The two keys are mutually exclusive.

Own-origin

Sign in your app, in code.

Your app owns the passkey's origin, so signing runs in-app. Set rpId. This is the path for wallet apps and anyone who owns their wallet's domain.

Shared-origin

Sign at the operator's origin.

The passkey lives elsewhere, so signing runs in a popup on web or an in-app browser tab on native. Set authOrigin. Your app never touches key material.

Sending is standard

You send with the tools you already use.

Avok announces an EIP-1193 provider over EIP-6963 and registers a Solana Wallet Standard wallet. wagmi, viem, and @solana/wallet-adapter discover it like any other wallet.

import { createAvokClient, createOwnOriginConnection } from "@avokjs/core";

const client = createAvokClient(
  { connection: createOwnOriginConnection({ rpId: "example.com" }) },
  { name: "Example Wallet", rdns: "com.example.wallet" },
);

const account = await client.create();       // runs the passkey ceremony
const provider = client.getEip1193Provider();
import { createWalletClient, custom, parseEther } from "viem";

const provider = client.getEip1193Provider();
const wallet = createWalletClient({ transport: custom(provider) });

const [address] = await wallet.requestAddresses();     // eth_requestAccounts
const hash = await wallet.sendTransaction({
  account: address,
  to: "0xRecipient",
  value: parseEther("0.01"),
});                                     // eth_sendTransaction
// Bring your own bundler and paymaster. The user pays the fee in a token.
const client = createAvokClient(
  {
    connection,
    bundlerUrl: "https://your-bundler.example",     // ERC-4337, EntryPoint v0.8
    paymasterUrl: "https://your-paymaster.example", // ERC-7677
    requireSponsorship: true,
  },
  wallet,
);

const provider = client.getEip1193Provider();
const { id } = await provider.request({
  method: "wallet_sendCalls",
  params: [{
    calls: [{ to: "0xRecipient", value: "0x0", data: "0x" }],
    capabilities: { paymasterService: { url: "https://your-paymaster.example", context: { token: USDC } } },
  }],
});

There is no useSend hook. Sending and signing go through the announced provider, so your transaction code is the same code you would write for any wallet.

Bring your own infrastructure

No defaults. Each endpoint is a trust boundary you choose.

Avok ships no third-party RPC, bundler, paymaster, or Kora endpoint, and never will. Supply the ones you trust; sponsored sends let users pay fees in a token.

RPC
Your provider, your proxy, or a URL your user supplies. Per chain.
Bundler + Paymaster
ERC-4337 and ERC-7677 for sponsored EVM sends, together.
Kora
One endpoint for sponsored Solana sends: fee payer and submitter.
Identity
The announced wallet is the operator's brand, never Avok's.

Chains

Two rails, one wallet.

EVM wallets are smart EOAs through EIP-7702. Solana is a first-class second rail. You pick a chain per send from the registry.
